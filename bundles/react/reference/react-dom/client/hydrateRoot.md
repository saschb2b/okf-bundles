# Reference

```js
import { hydrateRoot } from 'react-dom/client';

const domNode = document.getElementById('root');
const root = hydrateRoot(domNode, reactNode, options?);
```

Attaches React to HTML previously generated by [`react-dom/server`](/reference/react-dom/server/renderToPipeableStream.md), turning the static snapshot into an interactive app. For client-only rendering with no existing HTML, use [`createRoot`](/reference/react-dom/client/createRoot.md) instead.

## Parameters

- `domNode`: a DOM element rendered as the root on the server.
- `reactNode`: the React node used to render the existing HTML, usually JSX like `<App />` produced on the server by a method such as `renderToPipeableStream(<App />)`.
- optional `options`: an object:
  - optional `onCaughtError`: callback when React catches an error in an Error Boundary. Receives the `error` and an `errorInfo` with a `componentStack`.
  - optional `onUncaughtError`: callback when an error is thrown and not caught by an Error Boundary. Receives the `error` and an `errorInfo` with a `componentStack`.
  - optional `onRecoverableError`: callback when React automatically recovers from an error. Receives the `error` and an `errorInfo` with a `componentStack`. Some errors include the original cause as `error.cause`.
  - optional `identifierPrefix`: a string prefix for IDs generated by [`useId`](/reference/react/hooks/useId.md). Must match the prefix used on the server.

## Returns

An object with two methods: `render` and `unmount`.

# Root methods

## `root.render(reactNode)`

Updates the component inside the hydrated root. Returns `undefined`. If called before hydration finishes, React clears the server HTML and switches the whole root to client rendering.

## `root.unmount()`

Destroys the rendered tree, unmounts all components, and detaches React from the root DOM node, removing event handlers and state. Takes no parameters and returns `undefined`. After unmounting you cannot call `render` again on that root.

# Usage

## Hydrate server-rendered HTML

Call it once at startup. The React tree passed must produce the same output as the server did, or hydration errors result.

```js
hydrateRoot(document.getElementById('root'), <App />);
```

Common mismatch causes: extra whitespace around the root HTML, `typeof window !== 'undefined'` checks, browser-only APIs like `window.matchMedia` in render, or different data on server and client.

## Hydrate an entire document

Apps that render the whole document, including `<html>`, pass the `document` global.

```js
hydrateRoot(document, <App />);
```

## Suppress unavoidable mismatches

For a single element whose content unavoidably differs (for example a timestamp), set `suppressHydrationWarning={true}`. It works one level deep and is an escape hatch; React does not patch mismatched text.

## Handle different client and server content

For intentional differences, do a two-pass render: read an `isClient` state variable set to `true` in an [`useEffect`](/reference/react/hooks/useEffect.md), so the first pass matches the server and a second pass runs after hydration. This makes hydration slower.

## Update and error logging

Calling `render` after hydration is uncommon; usually you update state inside components. Pass `onCaughtError`, `onUncaughtError`, and `onRecoverableError` for custom error reporting.

# Caveats

- Expects the rendered content to be identical to the server-rendered content. Treat mismatches as bugs and fix them.
- In development React warns about mismatches; attribute differences are not guaranteed to be patched, for performance reasons.
- You usually have only one `hydrateRoot` call, often done by your framework.
- Not supported for client-rendered apps with no existing HTML. Use [`createRoot`](/reference/react-dom/client/createRoot.md).

# Troubleshooting

- "You passed a second argument to root.render": pass options to `hydrateRoot`, not `root.render`.
